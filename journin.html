我們最終在三週內成功交付，獲得業師 Design Review 回饋，最終順利取得完課證書。

背景

參與 AAPD UI 實戰營所執行的團隊專案。實戰營著重在提升 UI 設計能力以及與業界接軌能力。 我與其他五位參加者隨機分配為一組，主題上選擇了旅遊訂房。依循實戰營提供的專案範圍，在三週內完成必要項目的產出。 經歷前期研究，小組專注於背包客這一特定使用者群體，我們在 App 集成交流、探索和放鬆三個關鍵元素，我們將確保 App 的介面簡潔舒適，提供流暢的使用體驗。

成果

旅人間更多的探索與交流

可以看見他人的旅遊心情與照片，提供更多旅遊的想像與動機，也增進與他人的交流，是觸發訂房動機的關鍵因素。

更符合獨旅需求的篩選條件

在床型、設施、周邊環境等篩選條件，我將符合獨旅者需求的資訊層級提高。

專案範圍

由 AAPD 實戰營所提供，需在三週內完成下方使用者故事 (User story) 與必要項目 (Must Have)。

使用者故事 (User Story)

  • 我期望能在「訂房首頁」設定住宿條件（日期、地點、人數等）並搜尋，讓我能找到合需求的房源。
  • 我期望能在「搜尋結果頁」瀏覽不同房源的照片、價格、評分和評論，讓我能做出初步決策。
  • 我期望能在「房源資訊頁」查看房源的詳細資訊，包括全部照片、設施和附加服務，讓我能判斷是否要下訂此房源。

必要項目 (Must Have)

  • UI Mockups
  • UI Flow（ 包含流程與工程規格標註 ）
  • Figma 設計文件交付（ 包含完整的檔案封面與完稿頁面 ）
挑戰

如何創造市場區隔 ?

在開始製作前我們自問 : 目標使用者是誰? 因為在市場上已經有很多滿足訂房需求的服務，且產品已經相當成熟，我們不希望成果只是「換了視覺風格的 airbnb 」，我們該如何創造市場區隔以及發現尚未被滿足的使用者需求?

六位剛認識的 UI Designer 如何在短時間完成專案 ?

來到實戰營之前我們互不認識、沒有合作經驗。彼此間有很多不同的地方 : 設計風格、設計流程、身處的設計師階段，在職或是待業中...等等，我們如何克服這些差異，在每一次的討論中達成共識，最後完成團隊專案 ?
而團隊合作是我參與實戰營的主要原因之一，因為現階段在職的設計團隊是兩位設計師 ( 包含自己 )，所以希望可以參與多人協作的專案，實際了解多人開發的分工與設計過程。

研究

我們挑選六個直接競品做競品分析，分析維度著重在主要客群、房源類型與平台特色，我主要負責 Booking.com 這個品牌。發現除了基本的的訂房功能，每個產品都會因應客群或是品牌定位，延伸其他附加功能，例如套票行程、活動資訊等。透過競品分析，我們發現 : 房型與優先選項大多以家庭、雙人以上為主。

我進一步調查市場趨勢，發現後疫情時代，享受獨自旅遊已經成為一股風氣，根據外媒報導，去年在 Google 上，搜尋一人遊的人數，相較五 年前是幾乎翻了一倍，而過去三年在 TikTok 上，標註一人遊的內容，更是成長了近十倍。

設計假設

對背包客來說，重要的是甚麼 ?

我們認為背包客與一般旅遊者最大的是需求差異，一般旅遊者的需求是放鬆、吃些美食。觀光為導向。但背包客除了想放鬆之餘更多的是透過旅遊做向內的探索，以及找尋與和世界多一些連結感。為此，我們假設一名旅遊的背包客，有一款 App 可以讓他在旅遊前以及旅程中進行探索、與其他使用者交流，並完成訂房，達成這趟獨旅。

使用者研究

獨旅在乎什麼?

我們透過訪問身邊有過獨旅經驗者，他們在旅遊前、旅遊中開啟訂房 App 的情境、需求與感受，我們發現以下幾個共同點 :

洞察 1 : 深度交流

在旅行過程中大部分為一個人，而他們的目的主要為與陌生人交流、了解當地特色文化、放鬆追尋自我等。

洞察 2 : 地圖與訂房平台切換

部分使用者偏好未知的彈性旅程，視當天旅遊到哪裡，再搜尋附近適合的住宿。因此需要花大量時間在地圖與訂房平台切換。

洞察 3 : 即時與住宿方聯繫

背包客更容易有說走就走的旅遊動機，也因為彈性旅程，希望可以快速與住宿方聯繫，以便完成訂房。

學習與反思
  1. 產品思維的重要性

    製作專案的過程以及參與得獎組別的專案發表，讓我理解到產品思維對於介面設計有著非常重要的影響。產品思維強調將使用者需求、業務目標和技術限制融合在一起，要求設計師深入了解使用者面臨的問題和需求，並提供解決方案。介面設計應該將解決問題作為首要目標，通過直觀的設計和功能來解決使用者的痛點。

  2. 下一步 : 易用性測試與驗證假設

    礙於專案時程與要求，產品原型出來後沒有經過易用性測試與驗證假設。但以產品的角度來看，下一步需收集使用者反饋來驗證假設與解決方法是否有效，並根據這些反饋來調整和改進設計。這有助於確保設計符合使用者需求並持續提升使用者體驗。

  3. 確立產品核心目標的重要性

    製作過程中我們常會在不同解決方案中難以下決定，後來透過詢問業師與前輩，理解到其實每個方法都可行，最重要的是，哪一個是最符合品牌核心精神的方法。回溯到品牌的 Why 作為判斷的依據。若這麼做還是很難下決定，團隊間就要討論是否是核心精神不夠明確或是問題定義的不夠清楚。

團隊合作
短時間如何建立團隊默契

因為時間相當緊迫，在確認題目後我們很快就投入討論與製作。但大家來到這個活動的目標跟做事風格都不一樣，在討論越難越取得共識的結果是會議時間越來越長。在某次線上討論即將結束時，組員 Cherie 提議分享彼此對於目前的狀態與感受，任何內容都可以，也可以選擇不表達，只有一個規則是不要對他人的發言給予任何評價或回覆。透過大家很勇敢且信任對方的陳述想法，我們理解彼此在意的地方，參與活動的預期收穫等等。這些讓我們找尋更適合的協作模式、也更加了解彼此。這讓我了解到，做事之餘，提供一個安全的表達機會與環境也同等重要。

短時間如何建立團隊默契

時間緊迫加上大部分組員白天都有正職，會議時間通常都是晚上開始，也經常超過午夜結束。除了壓縮到休息時間，另一方面討論時間也會壓縮到實作時間。因此我們開始會對每次會議做簡單的覆盤，覆盤方向聚焦在 : 下次怎麼做可以更有效的討論與節省時間?
後來我們會固定用下面這些方式來改善 :

  • 事先擬定會議流程與每個環節預估時間，發在群組讓每個人知道。
  • 在會議過程中做會議紀錄，會議後發給大家。
  • 會議記錄中明確列出下一步有哪些項目與對應的負責人。
  • 每個會議尾聲做簡單的覆盤與心得交流，在做事之餘也要照顧到大家的狀態與想法，這種反思和改進的做法有助於提升會議效率和合作品質。
